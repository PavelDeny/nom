# ORM Responsibilities — Краткие правила использования ORM в проекте

Цель: единообразно определить, за что отвечает TypeORM, а за что — Mongoose, чтобы избежать рассинхронизации схем, дублирования индексов и путаницы в кодовой базе.

## Основное правило
- **TypeORM** — основной ORM. По умолчанию все новые сущности создаются через TypeORM. TypeORM используется для основных CRUD-операций, фильтрации, пагинации и работы с большим объёмом данных (например, Users, Spaces, Units, Bookings).
- **Mongoose** — инструмент для специальных задач:
  - тяжёлые агрегации ($geoNear, $group, $lookup и т.п.),
  - bulk-инсерты / массовые обновления (insertMany, bulkWrite),
  - экстренные обходы проблем (редкие кейсы, когда TypeORM ведёт себя капризно).

## Правила и процесс
1. Новая сущность — по умолчанию в TypeORM. Для исключения (использовать Mongoose) — в PR указывать причину: почему Mongoose предпочтительнее (например, выраженная агрегация).
2. Для коллекций, которые используются и в TypeORM и в Mongoose, гарантировать синхронность индексов и полей (описать и в entity, и в schema).
3. Аггрегации и bulk-операции организовывать в отдельных сервисах (например, `src/lib/services/searchService.ts` или `src/lib/services/bulkService.ts`), которые используют Mongoose напрямую.
4. Ввести `RepoFactory` (фабрику репозиториев), которая скрывает выбор реализации (TypeORM / Mongoose). Сервисы должны зависеть от абстракции `IRepo<T>`, а не от конкретного ORM.
5. Все изменения схем/индексов документировать и применять на проде через миграции или отдельные DBA-скрипты.
6. Транзакции: использовать MongoDB transactions только в replica-set; для меж-коллекционных операций предпочитать локальную семантику (raw session) или eventual-consistency паттерны.
7. Добавить тесты для основных агрегаций и bulk-операций.

## Быстрые подсказки
- Для агрегаций используем Mongoose `.aggregate(pipeline).exec()`.
- Для CRUD и фильтраций — TypeORM `repo.findAndCount()` / `findOne()` / `save()`.
